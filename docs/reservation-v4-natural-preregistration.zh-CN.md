# Reservation-v4 Natural 预注册

## 实验定位

本实验使用一个自然拆分的“取消预约并恢复库存”维护任务，验证在非字段定向设计下，结构化 handoff 是否影响后续 Agent 的正确性与执行成本。原始 issue 先确定业务验收标准，再由 TASK-001 的真实代码变更自然生成六字段 handoff；隐藏测试不按字段命名，也不为每个字段刻意设置一一对应的失败探针。

## 固定条件

- Agent：Codex，模型 `gpt-5.5`，推理强度保持一致。
- 每个 run 使用新对话和全新工作区。
- 后续 Agent 只能读取该 run 的过滤后 handoff 与工作区源码。
- 每个 run 最多 3 个 attempt；成功即停止。
- 每次 attempt 必须归档完整工具 Trace，并通过删除字段零命中的泄漏审计。
- 原始 issue、完整 handoff、隐藏测试和其他 run 不进入 Agent 工作区。

## 预注册假设

### H1：unfinished_items 与最终正确性

相对 `A_full`，`E_no_unfinished_items` 的首轮成功率和三轮内最终成功率更低，隐藏测试最终通过率更低，返工次数更高。

主要指标：首轮成功率、最终成功率、最终隐藏测试通过率。辅助指标：返工次数、接口猜测、无效编辑、总 Token。

### H2：verification 与执行效率

相对 `A_full`，`D_no_verification` 的总 Token、测试执行次数和返工次数更高，首轮成功率更低。

主要指标：总 Token、返工次数、首轮成功率。辅助指标：读文件、搜索、重复搜索、测试、接口猜测和无效编辑次数。

### H3：无 handoff 与仓库重新理解成本

相对 `A_full`，`H_no_handoff` 的读文件数、搜索次数、接口猜测、修改文件范围和总 Token 更高，最终正确性不高于完整组。

H3 为机制与负对照假设，不解释为任何单字段的普遍因果效应。

## Pilot

先运行 `A_full`、`D_no_verification`、`E_no_unfinished_items`、`H_no_handoff` 各 1 次。预注册随机顺序见 `reservation-v4-natural-run-plan.json`。只有满足以下条件才扩展重复：

1. `A_full` 能在三轮内通过 10/10 隐藏测试；
2. 四个 run 的泄漏审计均为 `passed`；
3. Trace 均完整且事件表可生成；
4. 隐藏测试失败不是基础设施错误；
5. 至少一个消融组相对完整组出现正确性或执行成本差异。

Pilot 通过后，四组各追加至至少 5 次；随后再决定是否运行 B、C、F、G 的完整 leave-one-out。

## 指标

- 正确性：隐藏测试通过数 / 10。
- 公开测试通过率：公开通过数 / 公开总数。
- 首轮成功率：attempt 1 是否公开与隐藏测试全通过。
- 最终成功率：最多 3 轮内是否全通过。
- Token：同一 run 全部 attempt 的 Codex 会话累计 token，按会话快照去重。
- 返工：`attempts - 1`，达到上限仍失败时记为右截断值 2。
- 人工干预：额外解释、纠错或方向建议的次数和分钟数；机械复制 Prompt、提交输出不计。
- Trace 事件：读文件、搜索、重复搜索、测试、编辑、接口猜测、无效编辑、回滚。
- 修改范围：唯一变更文件数及是否触碰 TASK-001 已完成模块。
- 泄漏审计：删字段内容和 canary 在全部可见输入、输出及工具 Trace 中的命中数必须为 0。

## 解释边界

该任务仍是单仓库、单领域任务。单次 pilot 只用于验证自然任务管线和效应方向，不用于宣称字段具有普遍重要性。正式结论至少基于每组 5 次重复，并需要后续不同自然任务复现。
