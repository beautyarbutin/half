# 用户故事

1. 我有多个 agent（包括 Claude Code、Codex、Copilot CLI、Minimax 等），这些 agent 大部分都是 OAuth 方式登录的。

2. 这个系统是一个 web 系统，我打开这个 web 系统后，点击创建 project，这时需要我填入以下信息：
   1. 选择本次任务需要哪几个 agent 参与协同（可选择的 agent 列表来源于"agent 一览页面"，每个 agent 至少包括名字、使用的模型、在任务里的别名、擅长的领域等信息）
   2. 本次任务的说明

3. 在我输入了项目说明后，系统会自动创建一个 git 仓库（可以是 GitHub 或其他的），用于本次项目协同，项目的说明会马上被写到这个 git 的一个页面。

4. 然后我进入本项目说明详情页，在这个页面里，最醒目的内容是告诉我下一步应该做什么的提示。这个时候我看到的是下一步工作是制定项目的工作计划，并提供了可一键复制的 prompt。

5. 于是我一键复制了 prompt，并把这个 prompt 手工发给我选择的一个或多个 agent，被选中的 agent 开始制定项目的工作计划。如果选择了多个 agent，每个 agent 需要独立完成。当我把 prompt 复制给所有的 agent 后，我还需要在页面上点一下"已派发"的按钮，启动一个负责轮询的脚本程序后台运行。

6. 在这个详情页面里，我可以看到当前任务的进度，这个时候显示是 agent 在制定工作计划，有一块区域我可以看到完成情况：如果已经完成了会显示绿灯，如果正在进行中，会显示红灯。

   > **关于任务状态检测的说明：** 由于 agent 是 OAuth 方式登录，我无法通过 API 获取任务是否完成的状态。这里使用的一个技巧：之前复制给 agent 的 prompt 里面有一个任务码（可能是随机生成的一串字符），当这个 agent 完成任务后，prompt 会要求它把结果写到 git 库的某个文件里，并在这个文件里加入这个任务码和它自己的名字。同时，之前点击"已派发"按钮后启动的脚本程序会定时轮询 git 库，当它发现某个文件里出现了任务码时，就知道某个 agent 完成了某项工作，从而把状态标为绿灯。

7. 当所有 agent 执行的任务都变为绿灯时，当前制定工作计划的任务就完成了。这时我在项目详情页会看到建议的下一步工作是工作计划评审，当然我觉得无需这一步或者人工来确定工作计划就行，也可以直接跳过这个工作进入到再下一步。如果我选择进行工作计划评审，那么就需要我选择一个或多个 agent 对其中的一个或多个工作计划进行评审，同样的，页面上提供了 prompt 供我一键复制，可以直接手工粘贴给某个 agent 执行，任务状态跟进的逻辑同之前。

8. 当所有的工作计划评审结束后，提示的下一步工作是确定最终工作计划，这一步我可以选择其中一个模型来进行，也可以人工进行。逻辑跟之前都是一样的。

9. 到了这个时候，我已经有了定稿的工作计划，工作计划里有一个章节的内容是具体的执行方案（根据我选定的 agent 数量和类型，已经进行了分工）。在项目详情页面上，已经根据执行方案，显示了参与工作的 agent 及其对应的 prompt，我只要逐个把 prompt 粘贴给对应的 agent 去执行就可以。跟之前一样，我可以在这个页面看到各个 agent 执行的情况。

10. 不可避免的，有些任务是串行的，当前序任务完成时，页面会提醒我接下来要把某个 prompt 发给某个 agent，我看到这个提示后当然会尽快操作。直到最后，所有的任务都完成，这个时候我可以在项目详情页看到一个执行报告。

11. **用量页面：** 在这个页面，我可以看到每个 agent 的实时用量情况。如果已经触发限额了，用红灯提醒，并有倒计时说明多久之后能恢复。

12. **Agent 一览页面：** 在这个页面，我能够看到我掌握的所有模型信息，包括每种模型的名称、账号订阅类型、到期时间、擅长领域、运行在哪台服务器上、对应的服务器登录用户等。

13. **服务器与用户一览页面：** 在这个页面，我可以看到各个 agent 运行的服务器的信息：
    1. 每个服务器的信息包括服务器的 IP 地址、所在场所（如阿里云、腾讯云、公司机房等）
    2. 每个服务器上的每个用户的信息包括用户名、运行的 agent、SSH public key 等

14. **SSH Key 管理页面：** 这个页面我可以浏览和管理每一个 SSH key。

15. **注意点：** 对于每一台机器的每个用户而言，每种 agent 只登录一个账号，这样可以避免频繁切换号码。
